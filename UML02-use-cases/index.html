UML 2 - Cas d'utilisation

Objectifs de la leçon

Savoir lire et rédiger des cas d'utilisation.

Introduction

Les use cases servent à documenter les fonctions d'une application.

Ils ont été introduites par Ivar Jacobson, au début des années 90, décrite dans le livre Object-Oriented Software Engineering. A Use Case Driven Approach, Ivar Jacobson, Addison-Wesley, 1992.

Ils peuvent avoir différents rôles comme

  1. documenter les exigences fonctionnelles d'un système informatique
    2. ,
  2. documenter les fonctions d'un système informatique dans le cadre d'un dossier d'analyse
  3. organiser l'implémentation
  4. servir de support pour des discussions, des communications, des workshops, ...

Tous ces rôles sont semblables car le but est le même, avoir un plan des fonctionnalités du système.

Dans le cadre du dossier d'analyse, les use cases serviront à analyser les fonctions de l'application, soit dans notre modélisation:

position du use case dans le contexte du dossier d'analyse.

Rappelons cependant que ce diagramme est un modèle, et qu'en réalité, il y a beaucoup d'autres manières de procéder.

Description

Qu'est-ce qu'un use cases?

C'est un cas d'utilisation du système.

Nous regardons le système du point de vue des acteurs, et nous nous demandons comment ils utilisent le système.

Le use case sert à capturer cette information.

Les use cases ne se réduisent pas aux diagrammes de use cases, bien au contraire. Le use case sert à capturer l'usage du système. En général, trois outils sont utilisés pour documenter les use cases:

  • le diagramme de use cases
  • la description textuelle d'un use case
  • le diagramme de séquence système

Illustrons maintenant chacun de ses outils.

diagramme de use cases

Le diagramme sert à illustrer l'ensemble des fonctionnalités du système, par exemple:

sparx systems, Use case diagram

Nous trouvons sur ce diagramme cinq symboles:

  • les acteurs, Products Manager, Chief Editor, On-line Customer et Customer Support Officer
  • les use cases, Add New Titles, Manage Publishers, Manage Titles, Create Orders et List Stock Levels
  • les associations entre acteurs et use cases
  • le système (ou l'application), Online Bookstore
  • une note Use case diagrams ...

Seuls les acteurs, use cases et associations sont obligatoires, cependant il est rassurant de voir les limites de l'application en plaçant le rectangle autour des use cases. Cette information est redondante car tous les use cases que nous listons ici font partie de l'application.

Les notes sont bien venues, mais nous devons faire attention à ne pas surcharger le diagramme.

Le diagramme ci-dessus est une bonne illustration. Notons que les use cases sont pour la plupart de haut niveau d'abstraction. Par exemple Manage Publishers sert à gérer les éditeurs, il y aura donc en général quatre opérations, créer un éditeur, consulter un éditeur, mettre à jour un éditeur et supprimer un éditeur (les fameux CRUD - Cread Read Upadate Delete).

description textuelle

Il est très utile de décrire textuellement un use case, et il existe de nombreux patrons (templates) que nous pouvons utiliser.

Prenons l'exemple d'une application pour gérer les demandes de congés (voir exercice 2 - demande de congés ). Le diagramme de use cases est:

La documentation textuelle du use case demande congé pourrait être:

Demande congé

Description

En tant que travailleur, je veux pouvoir demander un congé, ainsi je peux planifier mes vacances, mon temps libre.

Acteurs
  • Travailleur: acteur principal
  • Manager: acteur secondaire (il veut être informé)
Préconditions

Le travalleur doit être authentifié.

Scénario happy flow

  1. Le travailleur introduit les informations relatives à la demande de congé.

    La demande de congés contient les informations suivantes

    • son matricule
    • son nom
    • son prénom
    • la date de début de congé
    • la date de fin de congé
    • le type de congé (en chosissant dans une liste proposée par l'application)
    • optionnellement, il peut contenir un commentaire
  2. L'application envoie une notification au travailleur contenant l'information relative à la demande de congé (cf ci-dessus).
  3. L'application envoie une notification au manager du trailleur contenant l'information relative à la demande congé (cf ci-dessus).
Post conditions
  • Le congé a été enregistrée dans la base de données
  • L'état du congé est mis à "REQUESTED"
  • Une email de confirmation a été envoyé au travailleur
  • Un email d'information a été envoyé au manager
Scénario "boss"

Un cas particulier concerne le patron, celui qui n'a pas de chef, celui qui est au sommet de la hiérarchie de l'entreprise.

Dans ce cas, il n'y a pas d'acteur secondaire (Manager), aucun mail d'information n'est envoyé et le congé passe directement à l'état approbation par les ressources humaine:

  1. Le boss (Travailleur) introduit les informations relatives à la demande de congé (voir scénario précédent)
  2. L'application envoie une notification de confirmation au boss
Post conditions
  • Le congé est enregistré dans la base de données
  • L'état du congé est mis à "APPROVED_MANAGER
  • Un email de confirmation a été envoyé au boss

Nous avons les informations suivantes:

  • Le nom du use case, Demande congé
  • Une description courte
  • La liste des acteurs impliqués avec indication "principal/primaire" ou "secondaire"
  • Les préconditions, c'est-à-dire les conditions nécessaires pour que ce use cases puisse être exécuté
  • Les scénarios, c'est-à-dire une description des échanges entre les acteurs et le système, c'est la partie la plus importante de la description. Il y a généralement plusieurs scénarios:
    • le scénario principal (ou happy flow) qui décrit le fonctionnement attendu;
    • des alternatives;
    • des scénarios avec erreur.
    Il est essentiel de décrire le flux de données dans les scénario (répondre à la question quelles données sont échangées entre les acteurs et l'application);
  • Les postconditions, décrivent ce qui a changé dans l'état du système.

Nous pouvons documenter nos use cases à l'aide d'un diagramme de séquences dit "système", c'est-à-dire un diagramme de séquences où le système est réprésenté comme une boîte noire, voici un exemple:

Le diagramme visualise les échanges entre le système et les acteurs dans un ordre chronologique de haut en bas.

Le diagramme de séquences système est un bon outil de communication car il donne une représentation graphique des échanges ce qui est plus facile à lire qu'un long texte. Par contre, il est moins complet que la description textuelle.

Définitions

Définissons avec plus de précision chaque élément d'un use case.

actors

Un acteur est une personne ou un système informatique ou autre chose qui interragit avec notre application.

Nous différençons les acteurs primaires des acteurs secondaires.

Les acteurs primaires initient généralement la communication avec l'application et ils en tirent un bénéfice.

Les acteurs secondaires fournissent un service au use cases.

Voici un exemple:

San José State University - Requirement Modeling

Account Holder est un acteur primaire, il initie les use cases et il en est le bénéficaire.

Bank Server est un acteur secondaire, il n'initie pas les use cases, n'en tire pas de bénéfices et offre des services aux "System".

Notons que le nom donné à l'application System est ... peu explite, il aurait été préférable d'écrire ATM.

Notons aussi que nous éviterons d'utiliser l'icone "personne" pour représenter le Bank Server, un rectangle ou une image d'ordinateur est préférable.

use cases

Un use case est un cas d'utilisation, ce n'est pas une ellipse sur un diagramme de use cases!

Un cas d'utilisation est un concept abstrait et le diagramme de use case ne fait que documenter le cas d'utilisation en lui donnant un nom et en indiquant quels acteurs interragissent avec lui.

Le diagramme de use cases n'est qu'une vue graphique, agréable mais finalement assez pauvre des use cases. C'est un résumé.

Il est donc souvent nécessaire de documenter plus en détail le use case avec, par exemple:

  1. une description textuelle;
  2. un (des) diagrammes de séquence "système";
  3. un (des) diagrammes d'activités;
  4. des description d'interfaces, utilisateur ou autres;
  5. la description des acteurs, via des personna;
  6. des examples de réalisation des use cases;
  7. des spécifications de tests, formalisées par exemple avec gherkin;
  8. ...

Habituellement, les analystes utilisent les 2 premiers moyens. Personnellement, je trouve que les exemples et gherkin sont des approches très intéressantes car elles confrontent le use cases avec la réalité, au prix d'une perte de généricité.

La documentation via les interfaces utilisateurs est aussi très intéressante car c'est un moyen de communication facilement compréhensible par tous et une approche très motivante, comparer à du texte. Cependant, elle présente bien des risques dont le risque de se perdre dans des détails graphiques, le risque de faire deux choses à la fois (préciser l'interface graphique et définir ce que dois faire le système), le risque de décrire à un niveau d'abstraction trop bas (et donc, nous serons obligés de passer à un niveau d'abstraction supérieur pour réaliser un bon design de notre application).

Vérifications

Avons-nous collecté tous les use cases? Comment s'assurer que nous avons de bon use case? Quels sont caractéristiques d'un bon use case?

Brigit Penzenstadler

Dans sa vidéo Scenario-based models: Use cases , Brigit Penzenstadler donne la checkliste suivante:

  • Avons-nous trouvé tous les acteurs?
  • Avons-nous trouvé tous les rôles dans l'environnement de le système?
  • Y-a-t'il au moins un use case pour chaque acteur?
  • Avons-nous des acteurs qui jouent un rôle similaire dans leur relation avec le système? Si c'est le cas, nous les fusionnons en un seul acteur.
  • Est-il facile de comprendre ce que le système fait en relisant les use cases?
  • Le nom de chaque use case est-il unique, intuitif et auto-explicatif?
  • La description résumée donne-t-elle une image précise du use case?
  • Y-a-t'il au moins un acteur pour chaque use case?
  • Aucun use cases n'ont des comportements ou des flux d'événements similaires?
  • Avons-nous identifié tous les use cases?
  • Chaque use case est-il bien lié à une exigence fonctionnelle?
  • Tous les use cases de type CRUD (Create, Retrieve, Update, Delete) ont été supprimés (et remplacé par un use case "Gère")?

Craig Larman

Dans son livre "Applying UML and Patterns: An Introduction to Object-Oriented Analysis and Design and Iterative Development, 3rd Edition", Craig Larman écrit:

6.16 Principe: quels sont les tests qui permettent d'identifier des cas d'utilisation valides?

Lequel des cas d'utilisation suivants est-il valide?

  • négocier un contrat avec un fournisseur;
  • traiter les retours;
  • se connecter;
  • déplacer un pion sur un plateau de jeu.

Nous pourrions soutenir que ce sont tous des cas d'utilisation à différents niveaux, en fonction des limites du système, des acteurs et des objectifs. Nous évaluerons leur validité après avoir présenté les processus métider élémentaires.

Mais au lieu de nous demander de façon générale ce qu'est un cas d'utilisation valide, une question plus pertinente serait celle-ci: à quel niveau faut-il exprimer un cas d'utilisation de façon qu'il soit utile pour l'analyse des besoins de l'application?

Plusieurs règles empiriques existent, notamment les suivantes:

  • le test du patron
  • le test PME
  • le test de la taille
Le test du patron

Si votre patron vous demande: "A quoi avez-vous passé la journée ?". Vous répondez: "Je me suis connecté". Votre patron sera-t-il content?

Si ce n'est pas le cas, le cas d'utilisation échoue le test du patron, ce qui implique qu'il n'est pas fortement associé à l'obtention d'un résultat de valeur mesurable. C'est peut-être un cas d'utilisation de très bas niveau, mais ce niveau n'est pas adapté à notre analyse des besoins.

Le test PME

Un processus métier élémentaire ou PME est un terme issu du domaine de la réingénierie des processus de l'entreprise, dont voici la définition:

PME

Tâche effectuée par une personne en un lieu et un temps données, en réponse à un événement, et qui ajoute une valeur commerciale mesurable et laisse les données dans un état cohérent (par exemple Autoriser un crédit ou Tarifier une commande) [source perdue].

Concentrez-vous sur les cas d'utilisation qui reflètent des processus métier élémentaires.

Le test PME est similaire au test du patron, en particulier en termes de détermination d'une valeur mesurable.

Cet énoncé n'est pas à prendre absolument au pied de la lettre. Un cas d'utilisation n'est pas qualifiable de PME s'il nécessite deux personnes ou un déplacement ? Généralement, si. Mais l'espreit de la définition est tout à fait juste. Il ne s'agit pas d'une petite étape consistant par exemple à "supprimer un article" ou "imprimer un document". En revanche, le scénario principal comprendra souvent de cinq à dix étapes. Mais il ne demandera pas plusieurs journées ou plusieurs scéances de travail comme négocier un contrat avec un fournisseur": c'est une tâche qui s'effectue en une seule session et prendra probablement de cinq minutes à une heure. Comme cela est défini dans le Processus Unifié, le scénario met l'accent sur l'ajout d'une valeur observable et mesurable, et se termine lorsque le système et les données se trouve dans un état stable et cohérent.

Le test de la taille

Un cas d'utilisation est très rarement constitué d'une seule action ou d'une seule étape. Il comprend généralement de nombreuses étapes et sa version détaillée nécessitera souvent de trois à dix pages de texte. Lors de la modélisation des cas d'utilisation, une erreur courante consiste à considérer une étape appartenant à une suite d'étapes apparentées comme un cas d'utilisation en soi, par exemple définir un cas d'utilisation nommé Saisir un code d'article. Imaginez-en une version détaillée: sa taille réduite vous donne une indication de l'erreur.

Exemple: appliquer les test
  • Négocier un contrat avec un fournisseur.
    • Beaucoup plus vaste et plus long qu'un PME. Pourrait être utilisé comme cas d'utilisation métier, et non comme cas d'utilisation système.
  • Gérer les retours.
    • Le patron est content. Présente les caractéristiques d'un PME. La taille est correcte.
  • Se connecter.
    • Le patron n'est pas content si vous y avez passé la journée.
  • Déplacer un pion sur le plateau de jeu.
    • Ce n'est qu'une étape. La taille est trop réduites.
Violations raisonnables des tests

Même si la majorité des cas d'utilisation identifiés pour une application doivent réussir ces tests, les exceptions sont courantes.

Il est parfois utile de créer des cas d'utilisation correspondant à des sous-fonctions à l'intérieur d'n cas d'utilisation PME ordinaire. Une sous-tâche ou un extension, par exemple "payer à crédit", peut se répéter dans plusieurs cas de base. Il est alors préférable de l'isoler dans son propre cas d'utilisation, même si elle échoue au test PME ou au test de la taille, et de la lier à ces autres cas pour éviter de dupliquer du texte.

Authentifier un utilisateur ne passe peut-être pas le test du patron, mais c'est peut-être suffisemment comlexe pour justifier une analyse soigneuse, par exemple pour une fonctionnalité de "signature unique.

JMD

Je propose une approche où chaque use case est une transaction élémentaire avec l'application.

Une transaction élémentaire est déclenchée par un acteur. Nous parlerons d'événement.

Elle est caractérisé par un flux de données entre l'application. et au moins un acteur.

Elle est cohérente et laisse l'application dans un état cohérent.

Par exemple, Saisir son nom n'est pas une transaction élémentaire, elle est incomplète.

Par contre, Créer un compte est une transaction élémentaire.

L'approche proposée ici est de plus bas niveau que celle proposée par Brigit Penzendler et Craig Larman. En particulier, les opérations Mettre à jour son compte et bien un use case alors que Brigit exclut les opérations CRUD et Larman sous-entend que ce sont des use cases trop petits.

Le désavantage de cette approche est que nous avons beaucoup plus de use cases.

L'avantage est que chaque use case représente une petite fonction du système et cela rendra l'organisation de l'implémentation plus aisée.

Nous pouvons aussi vérifier la qualité de de notre diagramme de cas d'utilisation:

  1. avons-nous utiliser verbe + complément d'objet directe pour nommer le cas d'utilisation (bien sûr, il y a des exceptions);
  2. évitons les verbes "vides", comme faire, effectuer, metter, par exemple faire une réservation d'hôtel devient réserver un hôtel;
  3. utilisons le langage du métier;
  4. évitons les synonymes;
  5. un cas d'utilisation est généralement une suite d'opérations synchrones (effectuées sans interruption dans la majorité des cas);

Nous pouvons aussi vérifier la qualité de nos descriptions:

  1. décrivons-nous ce que font les acteurs avec l'application et rien d'autres ?
  2. décrivons-nous les informations échangées entre l'application et les acteurs ?
  3. ET signifie souvent que nous avons deux étapes;
  4. n'inventons pas des règles métiers sans les discuter avec les représentants du métier;
