Patrons

En analyse, nous rencontrons souvent les mêmes "motifs", les mêmes types d'organisation entre classes. Nous appelons ces modifs "patterns" ou "patrons" en français.

La référence dans ce domaine est certainement le livre de Martin Fowler, Analysis patterns , qui malheureusement date de 1996 et utilise des notations "pré-UML". Depuis Martin a publié différents pdf avec des versions UML des patterns, par exemple:

Association n2n

Nous rencontrons souvent des classes qui sont liées par des associations n à n, c'est-à-dire qu'un objet instance de la classe A peut référencer n objets instances de la classe B et qu'un objet de la classe B peut référencer n objets instances de la classe C.

Voyons cela par des exemples.

  • address

Arbre simple

Les structures en arbre sont très courantes dans le monde réel et elles tiennent une place de choix en informatique.

En analyse fonctionnelle, les arbres sont utilisés pour représenter des structures hiérarchiques, comme les organigrammes d'entreprise:

ADVALORIS - Les différents types d'organigrammes d'entreprise

Ou bien la taxonomie des espèces animales:

Kansas State University - Tree data structures

ou encore une structure hiérarchique des pages d'un site web:

Kreezalid - How to build an effective navigation for your online marketplace

le système de fichiers des ordinateurs:

Kansas State University - Tree data structures

Le vocabulaire associé au structure en arbre est le suivant:

  • root est la racine de l'arbre (la base)
  • node est un noeud de l'arbre, un noeud peut contenir d'autres noeuds et des feuilles
  • leaf est une feuille de l'arbre, une feuille est une extrémité de l'arbre, elle ne peut pas contenir d'autres noeuds ni d'autres feuilles
  • parent est un noeud parent d'autres noeuds ou feuilles
  • child est un noeud ou une feuille enfant d'un autre noeud, notons qu'une feuille est toujours l'enfant d'un noeud
stackoverflow - How to store Linked Tree with children in Relational Database?

Arbre complexe

Dans bien des cas nous aurons des arbres composés de deux sortes d'éléments, les branches et les feuilles.

Les branches peuvent contenir d'autres branches et des feuilles.

Les feuilles sont des points terminaux de l'arbre.

fenêtre 2

Remettons notre cas précédent sur l'atelier Premièrement, nous différençons les noeuds, appelés ici Container car ils contiennent d'autres objets. Ensuite, nous créons une classe spécifique pour le Text, le InputText et le Button.

Le diagramme de classe devient:

Cette organisation des classes est une nouvelle signature d'une struture en arbre, certe plus complexe.

Au sommet du diagramme, nous avons GraphicalObject qui est en italique, pour indiquer que cette classe est abstraite, c'est-à-dire qu'elle ne pourrait jamais être instanciée. En effet, nous ne voulons pas instancier des objets de type GraphicalObject car tous les objets seront soit du type Container, soit Text, soit InputText, soit Button.

Les classes Text, InputText et Button héritent de GraphicalObject, c'est-à-dire qu'ils sont des GraphicalObject. Ils ont chacun leurs caractéristiques et comportements mais ils sont tous des GraphicalObject.

La classe Container sera utilisée pour représenter la fenêtre, l'entête, le "main" et le pied de page, bref toutes les zones d'affichage qui contiendrons d'autres objets graphiques.

En plus d'hériter de GraphicalObject, Container a une association 1 to N avec GraphicalObject. Cette association est la signature d'une structure en arbre avec GraphicalObject qui représente n'importe quel noeud ou feuille de l'arbre, et Container qui représente un noeud, c'est à dire un élément de l'arbre qui peut contenir d'autres élément.

Nous notons que les classes Text, InputText et Button ne peuvent pas avoir le rôle parent, elles ne font qu'hériter du rôle de children à travers la classe GraphicalObject.

DDC

DDC ou demande de congés est l'application "modèle" de ce cours.

Le diagramme de classe pourrait être celui-ci:

Nous avons 4 classes:

  • Worker qui modélise un travailleur, nous avons considéré les attributs suivants:

    • number qui est le numéro de matricule du travailleur
    • firstname
    • lastname
    • email
    • phonae

  • Manager qui hérite de Worker

  • DayOff, le congé, qui a pour attributs:

    • start la date de début de congé
    • end la date de fin de congé

    • state l'état du congé, de type DayOffState qui liste tous les états que peut prendre un congé:

      • REQUESTED pour une demande qui vient d'être introduite
      • APPROVED_MANAGER lorsque la demande est approuvée par le manager
      • APPROVED lorsqu'elle est approuvée par tous
      • CONSUMED lorsque le travail a pris son congé
      • REFUSED_MANAGER lorsque la demande a été refusée par le manager
      • REFUSED_RH lorsque la demande a été refusée par le RH
      • CANCELLED lorsque la demande a été annulée
    • comment, un commentaire libre fournit par le demandeur

  • DayOffCategory, les catégories des congés, par exemple, "congé annuel", "congé de maternité", "congé de paternité", ...

Le Manager est un Worker, il hérite donc de Worker. Nous avons aussi une association 1-N entre Manager et Worker, avec l'héritage, c'est la signature d'une structure en arbre, celle de l'organigramme de l'entreprise.

Concernant les associations, nous avons:

  • Un Worker a zéro ou un Manager, celui qui a aucun manager est le grand patron
  • Un Manager a zéro ou plusieurs Worker sous sa responsabilité
  • Un Worker peut avoir zéro ou plusieurs DayOff
  • Un DayOff concerne un seul Worker
  • Un DayOff est d'une seule DayOffCategory
  • Une DayOffCategory est la catégorie de zéro ou plusieurs DayOff
